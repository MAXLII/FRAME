# AI 执行规则

## 1. 目的

本文档用于约束 AI 在本仓库中的执行方式。

当前约束内容包括：

- Git 提交规则
- 版本号递增规则
- `docs/ENGINEERING_DESIGN.md` 编写规则

## 2. 适用范围

本文档适用于本仓库后续代码、工程、构建、文档和配置相关工作。

## 2.1 工作区边界

AI 默认只在当前工作区内读取和修改文件。

如果任务需要修改当前工作区之外的代码、工程文件、脚本或文档，AI 必须在动手前再次向用户确认具体路径和修改意图。

即使用户在对话中提到了工作区外的路径，也不能直接写入该路径；只有在用户明确确认“可以修改该工作区外路径”之后，才允许进行文件修改。

该规则用于避免用户忘记切换进程、目录或仓库时，AI 误操作其他工程。

## 3. Git 提交规则

### 3.1 单次提交只表达一个主题

一次提交只包含一个清晰的变更主题，例如：

- 一个模块重构
- 一个工程目录拆分
- 一组构建脚本调整
- 一次文档更新
- 一组编辑器配置调整

不将多个无直接关系的改动混在同一笔提交中。

提交分批必须以实际代码内容为准。AI 在提交前需要查看 `git diff` 或等价的实际变更内容，按真实修改到的模块职责、接口行为、用户可见功能、构建配置和文档内容拆分提交。

不能只根据文件名、最近一次用户指令或聊天上下文粗略打包提交。即使多个改动发生在同一轮对话中，只要实际代码内容属于不同功能、不同页面、不同协议、不同配置或不同文档主题，就需要拆成多笔提交。

如果同一个文件内同时包含多个互不直接依赖的变更主题，应优先使用交互式暂存或等价方式按代码块分批提交；只有这些改动在代码行为上强依赖、无法合理拆开时，才可以放在同一笔提交中。

### 3.2 目录调整与功能改动分开提交

目录迁移、文件重命名、工程拆分，单独提交。

功能逻辑修改、接口行为修改，单独提交。

### 3.3 构建系统改动单独成批

`makefile`、`CMakeLists.txt`、`compile.bat`、`download.bat` 等构建相关文件，按构建主题单独提交，或与其直接服务的工程拆分一起提交。

### 3.4 文档与代码分开提交

`README.md`、`docs/` 下文档更新，单独提交。

如文档只是对本次代码改动进行同步说明，也作为单独一笔文档提交。

### 3.5 编辑器与工具配置分开提交

`.vscode/`、脚本工具、辅助配置文件，单独提交。

不与功能逻辑改动混在同一笔提交中。

### 3.6 新工程提交方式

当仓库中新增一个工程时，按以下方式分批提交：

1. 工程目录骨架
2. 构建系统接入
3. 模块代码接入
4. 文档与配置同步

例如新增 `gd32g553c/xx/` 或 `plecs/xx/` 工程时，不将目录、代码、构建、文档一次性混在同一笔提交中。

### 3.7 提交类型

本仓库提交信息使用简短类型前缀。

当前使用以下类型：

- `feat`：新增功能
- `fix`：缺陷修复
- `refactor`：重构、重命名、目录调整、工程拆分
- `build`：构建系统调整
- `docs`：文档更新
- `chore`：工具、配置、辅助性整理

### 3.8 提交信息写法

提交信息格式如下：

```text
<type>: <简短中文说明>
```

示例：

```text
refactor: 拆分 GD32 AC/DC 硬件工程
refactor: 重构故障与告警状态模块
docs: 更新工程目录与设计说明
chore: 拆分 VS Code C/C++ 工程配置
```

### 3.9 执行要求

- 提交前确认本批改动边界清晰
- 提交前确认未混入无关文件
- 提交后工作区保持可继续整理
- 多批提交时，按阅读和回溯友好的顺序提交

### 3.10 构建与调试相关改动继续细分

当同一轮工作同时涉及底层驱动、构建脚本、下载脚本、调试工程配置时，继续按职责拆分提交，不合并成一笔“构建下载调试配置”提交。

- 驱动实现或 BSP 功能接入，单独提交
- `makefile`、`CMakeLists.txt`、`compile.bat`、构建输出目录调整，单独提交
- `download.bat`、`.svd`、`.jdebug` 等下载与调试配置，单独提交
- 相关文档同步，单独提交

如同一文件同时包含多类职责调整，先按更细的职责边界整理后再提交。

### 3.11 版本号定义

本工程版本号格式为：

```text
a.b.c
```

各字段含义如下：

- `a`：目标用户版本。不同目标用户使用不同主版本号，例如 `1` 表示作者自用版本，`2` 表示 DR 使用版本。
- `b`：大功能版本。新增页面、对某个页面进行大规模重做、功能形态发生明显变化时递增。
- `c`：小功能与后端优化版本。后端逻辑修改、小功能补充、缺陷修复、体验优化、协议细节调整时递增。

版本号递增规则：

- `a` 不由 AI 自动修改，除非用户明确要求切换目标用户版本。
- 发生大功能更新时，`b + 1`，并将 `c` 归零。
- 发生小功能、后端优化或缺陷修复时，`c + 1`。
- 只有文档、注释、忽略文件或提交手册变化时，不主动修改软件版本号，除非用户明确要求。

### 3.12 用户要求“提交代码”时的固定流程

当用户表达“提交代码”“根据变更提交代码”“提交当前代码”等意图时，AI 必须先阅读本文件中的提交规则与版本号规则，再执行提交。

固定流程如下：

1. 查看当前工作区变更，确认是否存在用户未要求处理的无关文件。
2. 查看实际代码差异，按第 3 章提交规则和真实变更内容拆分提交边界。
3. 先提交当前功能、修复、文档或配置变更。
4. 根据第 3.11 节判断版本号递增方式。
5. 修改项目版本号定义文件。
6. 单独提交版本号变更。

提交要求：

- 当前变更提交和版本号提交必须分成两笔提交。
- 版本号提交信息使用 `chore: 更新版本号至 x.y.z`。
- 如果无法确定版本号定义位置，先在仓库内查找；仍无法确定时再询问用户。
- 如果当前变更同时包含大功能和小功能，以大功能规则为准，即 `b + 1` 且 `c = 0`。

## 4. 通信协议设计规则

### 4.1 不使用保留位

本工程新增通信协议结构体时，不主动添加 `reserved` 保留字段。

协议字段只包含当前真实使用的数据。

### 4.2 兼容方式

协议兼容通过帧中的 `data_len` 判断。

后续扩展协议时，只允许在结构体末尾新增字段，不在已有字段中间插入字段。

接收端解析结构体 payload 时，应使用协议帧中的 `data_len` 和本地代码中目标结构体的实际大小 `sizeof(local_struct)` 取小进行兼容：

```c
copy_len = MIN(data_len, sizeof(local_struct));
```

然后只拷贝或解析 `copy_len` 范围内的数据。不能简单地把某个固定 `data_len` 当作兼容判断条件。

旧代码接收新协议时，`data_len` 可能大于旧结构体大小，旧代码只拷贝旧结构体大小。

新代码接收旧协议时，`data_len` 可能小于新结构体大小，新代码只拷贝实际收到的数据，新增字段保持默认值。

### 4.3 字段顺序

已发布字段的顺序和含义保持稳定。

如需扩展，按“尾部追加字段”的方式新增。

### 4.4 ACK 判定原则

`ACK` 只表示对同一个 `cmd_set / cmd_word` 请求的应答。

如果上位机发送 `0x01 / 0x18`，下位机也通过 `0x01 / 0x18` 回包，则该回包应设置 `is_ack = 1`。

如果上位机发送某个指令后，下位机因为该指令触发了后续逻辑，并通过其他 `cmd_word` 主动上报数据，则这些不同指令的上报帧不应设置 `is_ack = 1`，应按普通上报帧处理。

设计新协议时，不能把“由某个请求间接触发”误判为 ACK。只有同一个指令的直接响应才是 ACK。

## 5. `docs/ENGINEERING_DESIGN.md` 编写规则

### 5.1 文档定位

`docs/ENGINEERING_DESIGN.md` 用于说明本工程当前目录和模块设计。

### 5.2 只写当前已有内容

文档内容基于当前仓库中已经存在的目录、文件和模块编写。

不写未来规划，不写建议方案，不写“后续准备怎么做”。

### 5.3 直接定义当前目录

文档中直接写“工程目录”“目录结构”“模块职责”。

不写“推荐目录”“建议采用”“推荐组织方式”等表达。

### 5.4 按模块职责说明

工程设计文档重点说明各目录、各模块当前的功能设计内容和职责划分。

不写成教程式、介绍式或教育式文档。

### 5.5 使用正向描述

文档描述目录和模块时，只说明“是什么”“做什么”。

不通过“它不是什么”“这一层不属于什么”这类反向表达来说明。

### 5.6 不写历史过程

文档只描述当前状态。

不写“原来叫做什么”“曾经放在哪里”“后来从什么改成什么”这类历史过程说明。

### 5.7 不写无必要示例

当目录职责已经表达清楚时，不额外插入“示例：”段落。

如需列出现有内容，直接列当前目录、当前文件或当前模块即可。

### 5.8 目录内容以实际仓库为准

目录树、模块列表、文件列表以当前仓库实际内容为准。

如某文件已删除、已迁移或当前不存在，不在文档中继续保留描述。

### 5.9 忽略文件不做说明

工程设计文档不说明忽略文件。

本地生成文件、日志文件、缓存文件、临时文件以及 `.gitignore` 中约定忽略的内容，不写入 `docs/ENGINEERING_DESIGN.md`。
